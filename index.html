What the user does

  • Taps Sign in with Apple or types an email + password.
  • Ticks two boxes: "I'm 18+" and "I accept Terms & Privacy." Both are required.
  • Picks a date of birth from a wheel picker — never types an age in a textbox.
  • Answers 8 to 15 onboarding questions that read like a chat with a thoughtful friend.
  • Adds photo, gender, bio, location, voice note.
  • Gets a final "Your avatar is ready" moment.
For engineers

What the system does

  • Auth: Apple OIDC or Supabase email/password creates a row in auth.users.
  • Trigger handle_new_user fans out and creates 6 child rows in one transaction.
  • DOB write hits a trigger that derives age and rejects anyone under 18 server-side.
  • Each intro turn hits the avatar-chat Edge Function in intro mode.
  • That function fires two parallel Claude calls — one for the next question, one for OCEAN extraction.
  • Every Claude round-trip lands in claude_chat_log for future Persona AI training.

Rule.
The onboarding chat is the only path that writes to personality_profiles. The avatar reflects the user's own words — never staff curation, never editorial defaults.
How a single intro turn becomes a personality update

Each intro turn fires two Claude calls in parallel:

① Question generator — the system prompt includes the running profile summary and the categories already covered. Returns the next question.

② Extractor — a separate Claude call reads the user's last answer and returns structured JSON: numeric deltas to each OCEAN dimension, plus communication style, humor, conflict style, attachment style, love languages, core values, interests, deal-breakers, and short memory facts.

The client merges deltas into personality_profiles (clamping each field to [0,1]) and inserts new personality_insights rows.

Both raw Claude calls are also written to claude_chat_log with token counts, latency, full prompt, and full response.

What compliance gates exist and why

App Store 1.1.4 / 1.3 (age verification). DOB is the source of truth. A trigger derives age on every write and rejects anyone under 18 with a 23514 check_violation error.

App Store 5.1.1 (privacy + ToS). Two required checkboxes at signup. Both link to live policy URLs. The "Create account" button is disabled until both are checked.

App Store 1.2 (user-generated content). Block, report, and account-delete are all live. Block enforcement is server-side in the discovery RPC — even if the client forgets to exclude blocked IDs, blocked profiles won't surface.

For engineers

What runs in the database

  • Client calls Postgres function nearby_candidates(p_user_id, lat, lon, ...).
  • Server applies a filter chain: not self, active, discovery enabled, age range, gender, within 100mi (Haversine), excluded IDs, not blocked either direction.
  • Compatibility scored client-side by the PersonaMatching SPM package against the OCEAN profile.
  • High-scoring pairs become connections rows in status=pending with score + insight + shared themes.
  • Each pending connection then triggers an avatar-to-avatar exchange (Phase 3).

Rule.

How compatibility is actually scored

Two OCEAN vectors (5-dimensional) are compared via cosine similarity, then weighted by domain factors: communication-style match, shared interests, complementary attachment styles, deal-breaker intersection.

The scorer lives in Packages/PersonaMatching and runs entirely on-device. No Claude call is needed for the match itself — Claude is reserved for the conversation generation in Phase 3.

High-score pairs (above an internal threshold) become connections. Marginal pairs are kept warm for the next day's batch.

How the conversation is generated

  • A new connections row spawns an avatar_conversations row with both avatar IDs.
  • A matching engine alternates calls to avatar-chat — once as A, then as B — passing each side's persona context.
  • Claude Haiku replies in-character, capped at 1–3 sentences with a lowercase-default tone.
  • Each reply writes to avatar_messages with a monotonic sequence_number.
  • Every Claude call lands in claude_chat_log.
  • After ~6–12 turns, a summary + topics_discussed are written back to avatar_conversations.

Rule.
The avatar speaks as the user's wingman, never as the user. It opens with curiosity, asks real questions, and never makes up facts about the human it represents.
Rule.
The system prompt enforces a 1–3 sentence cap per reply, lowercase-default tone, and US-only crisis-line copy (911 · 988 · 741741) if the conversation veers into safety territory.
What goes into the system prompt for each turn

Built fresh each call from buildChatSystemPrompt(ctx):

{
  avatar_name, user_name, personality_summary,
  conversation_topics, love_languages, core_values,
  communication_style, humor_style, recent_memory_facts
}

The system block is sent with cache_control: "ephemeral". Anthropic's prompt caching makes follow-up turns within a 5-minute window roughly 10× cheaper.

Rate limits and cost controls

Free tier: 150 Claude calls per user per day. Premium: 1000/day.

Enforced in avatar_chat_usage by the edge function before any Claude call fires. Cap exceeded returns 429 {error:"rate_limited"} and the client surfaces a friendly toast.

What runs underneath

  • iOS opens one WebSocket on launch, subscribes to realtime:public:messages filtered by conversation_id.
  • Sending a message: POST to messages table → trigger notify_on_new_message inserts a notifications row → Realtime broadcasts both rows.

How users stay safe

For engineers

How it's enforced

  • blocks table is read by the nearby_candidates RPC — server-side filter.
  • reports row inserted in pending; admins resolve via admin_actions audit log.
  • Account deletion calls the delete-account edge function → auth.admin.deleteUser → cascade deletes through every FK-linked table.
